# TM System Design

## Overview

Task-API is a Task Manager API built as a backend engineering project.

The goal is to keep the product small and clear while evolving the internal architecture.

## General Flow

`Router → Schema → [mapper when needed] → DTO → Service → Domain → Repository → PostgreSQL`

### Note:

`Write use cases are transaction controlled by the Service layer through UnitOfWork`

## Layer Responsibilities

- Router handles HTTP concerns
- Schema validates request and response data
- Service coordinates the use case
- Domain enforces business rules and state consistency
- Repository abstracts persistence and maps domain entities to the persistence layer
- Persistence stores application data in PostgreSQL

## Endpoints

Read flows:
- the service reads through the repository from PostgreSQL
- no transaction commit or rollback is needed

Write flows:
- the service controls the transaction
- on success, it commits
- on failure, it rolls back
- if no real change is detected, it does not save or commit

### POST /tasks

`Router → Schema → TaskInput [DTO] → Service.create_task() → [Domain] Task.create() → Repository.create_task() → PostgreSQL`

#### Note:

- new tasks are created in the domain without a persistent id
- the task id is generated by PostgreSQL during persistence
- the repository returns the persisted task with the generated id

### GET /tasks

`Router → Service.list_tasks() → Repository.list_tasks() → PostgreSQL`

### GET /tasks/{id}

`Router → Service.get_task() → Repository.get_task() → PostgreSQL`

If found:

`Service → return task`

If not found:

`Service → raise TaskNotFoundError`

### DELETE /tasks/{id}

`Router → Service.delete_task() → Repository.get_task()`

If not found:

`Service → raise TaskNotFoundError`

If found:

`Service → Repository.delete_task() → PostgreSQL`

On success:

`Service → commit transaction`

On failure:

`Service → roll back transaction`

### PUT /tasks/{id}

`Router → Schema → TaskInput [DTO] → Service.update_task() → Repository.get_task() → Domain Task.update()`

If changed:

`Service → Task.mark_updated() → Repository.save_task() → PostgreSQL`

If unchanged:

`Service → return current task without save or commit`

### PATCH /tasks/{id}

`Router → Schema → Mapper → PatchTaskInput [DTO] → Service.patch_task() → Repository.get_task() → resolve patch into full TaskInput target state → Domain Task.update()`

If changed:

`Service → Task.mark_updated() → Repository.save_task() → PostgreSQL`

If unchanged:

`Service → return current task without save or commit`

### PUT/PATCH Update Flow

#### Service

PUT and PATCH share the same service-level update path through `_apply_update(task, task_input)`.

- `PUT` already provides a full `TaskInput`
- `PATCH` provides `PatchTaskInput`, which the service resolves into a full `TaskInput`
- `_apply_update(...)` then delegates the state change to `Task.update(...)`

If the state really changes:

`Service → Task.mark_updated() → Repository.save_task() → PostgreSQL`

#### Domain

```text
Task.update(...)
├─ _ensure_editable()             → blocks updates for terminal tasks
├─ _validate_title()              → trims and validates title
├─ _validate_status_transition()  → validates status changes when needed
├─ _validate_blocked_transition() → validates blocked transition rules when needed
└─ _validate_final_state()        → validates final task state consistency
   ├─ _validate_due_date()        → validates due date rules
   └─ _validate_blocked_state()   → validates blocked state rules
```

## Core Domain

### Task
id · title · description · status · due_date · created_at · updated_at · is_blocked

#### Status

PENDING · IN_PROGRESS · COMPLETED · CANCELLED

#### Editable status

PENDING · IN_PROGRESS

#### Terminal status

COMPLETED · CANCELLED

#### Task Lifecycle

```text
   ┌────────────────────┐
   ▼                    │
PENDING ─────────▶ IN_PROGRESS ─────────▶ COMPLETED
   │                    │
   │                    └───────────────▶ CANCELLED
   │                                           ▲
   └───────────────────────────────────────────┘
```

Note:

- blocked tasks follow extra transition restrictions during update

#### Current Task Rules

- title is trimmed and cannot be empty
- title must be at least 3 characters long
- title must be at most 120 characters long
- IN_PROGRESS requires due_date
- COMPLETED requires due_date
- due_date cannot be in the past
- COMPLETED cannot remain blocked
- updates enforce extra rules for specific status and blocked-state combinations
- updates are fully validated before changes are applied
- tasks are saved only when the state really changes

## Persistence

- repository abstraction
- PostgreSQL persistence through SQLAlchemy
- explicit transaction handling through UnitOfWork
- in-memory repository retained for isolated testing

### Identity

- new tasks are created in the domain without a persistent id
- PostgreSQL generates the id during persistence
- the repository returns the persisted task with the generated id

## Database Bootstrap vs Schema Migrations

The project separates database bootstrap from schema evolution:

- database bootstrap is responsible for creating the database, roles, and grants
- Alembic is responsible for creating and evolving schema objects inside the existing database

Bootstrap prepares the PostgreSQL environment, but it does **not** create application tables.
Alembic must apply the current schema before the application starts using the database.

This keeps security and environment provisioning separate from application schema versioning.

## Direction

The project is evolving in small steps:

- keep domain rules explicit
- keep the architecture simple
- evolve the PostgreSQL persistence layer through migrations
- prepare the base for future features such as authentication and ownership
